# Kite 执行状态面板设计

- 日期：2026-03-26
- 状态：待评审
- 适用范围：`MessageList`、`ThoughtProcess`、`SkillCard`、`SubAgentCard`、执行态消息流、运行时事件映射

## 1. 背景

当前聊天区在 AI 执行期间，会把不同类型的执行信息拆成多块展示：

1. `ThoughtProcess`
2. `SkillCard`
3. `SubAgentCard`
4. 工具调用相关的日志行

这些信息虽然都在同一轮回复附近，但用户需要自己理解：

1. 当前 AI 是否真的在推进任务
2. 已经调用了哪些 `skill`
3. 是否调度了多个智能体
4. 当前执行到了哪一步
5. 为什么现在还没有正式输出

现有“思考中 / 思考过程”文案也过于抽象，不能准确表达“AI 正在执行任务”这一状态。对于非技术创业者和轻技术用户，这会带来明显的不确定感。

本次设计不再把该区域定义为“思考框”，而是定义为一个统一的 **执行状态面板**。

## 2. 问题定义

当前实现的主要问题：

1. **信息分散**
   - `thoughts / skills / subagents` 分散渲染，用户需要自己拼接上下文。
2. **标题抽象**
   - “思考中”“思考过程”不能直接回答“AI 现在在做什么”。
3. **默认不可见**
   - 运行中区域默认不是稳定展开态，用户感知不到持续推进。
4. **缺少固定摘要**
   - 用户看不到“已调度几个智能体、已加载几个 skills、当前步骤是什么”。
5. **阶段切换表达不清**
   - 多段执行时，旧阶段和新阶段容易以多个块堆叠，层级不明确。
6. **关闭时机不明确**
   - 执行态区域与正式输出的交接没有统一心智模型。

## 3. 设计目标

### 主目标

把执行期的所有关键状态收敛成一个统一面板，让用户在 AI 正式输出前，始终明确知道：

1. AI 正在执行任务
2. 有哪些智能体参与
3. 有哪些 `skill` 被加载
4. 当前执行到了哪一步
5. 刚刚完成了什么动作

### 成功标准

1. 运行中时，用户无需手动展开即可看到执行状态。
2. 用户在 1 秒内能看懂“当前 AI 在干什么”。
3. 面板不会无限拉长消息流，而是通过固定高度 + 内部滚动承载长日志。
4. 正式输出开始时，执行面板自动关闭，不与最终答案抢注意力。
5. 多智能体、多技能、多阶段执行时，界面仍只存在一个主面板。

### 非目标

以下内容不在本次范围内：

1. 公开展示完整原始推理全文
2. 暴露底层协议细节给普通用户
3. 改造最终 assistant 正文排版
4. 改造非执行期的普通消息卡片

## 4. 最终产品原则

### 4.1 执行优先于思考

用户需要看到的是“AI 正在执行什么”，不是抽象的“AI 正在想什么”。

### 4.2 一个面板优先于多个散卡片

执行期必须形成一个统一心智模型，而不是 `ThoughtProcess + SkillCard + SubAgentCard` 的机械拼接。

### 4.3 概览优先于流水账

面板必须先给出固定摘要，再给出滚动日志。用户不应该先读日志才能理解当前状态。

### 4.4 执行态与输出态分离

执行状态面板只服务于“AI 还没正式回答”的阶段；一旦正式输出开始，面板必须退出主舞台。

## 5. 方案结论

本次采用 **统一执行状态面板（ExecutionPanel）** 方案，不采用仅修改文案或仅默认展开的轻改方案。

### 选择该方案的原因

1. 用户诉求不是“改成默认展开”这么简单，而是“明确知道 AI 在干什么”。
2. 现有分散卡片结构无法稳定表达“智能体、skills、工具、阶段”的整体关系。
3. 统一面板可以同时解决默认可见、摘要固定、自动滚动、阶段切换和输出关闭五类问题。

## 6. 信息架构

执行状态面板由四层组成：

### 6.1 顶部标题区

固定回答三个问题：

1. 现在是什么状态
2. 当前在做什么
3. 有多少参与者

推荐结构：

1. 主标题：`AI 正在执行任务`
2. 次级摘要：`已调度 2 个智能体 · 已加载 3 个 skills`
3. 当前步骤：`当前步骤：检查 PDF 内容结构`
4. 状态徽标：`执行中 / 已完成 / 执行失败`

### 6.2 固定参与者区

持续显示本轮已参与的执行主体：

1. `Agents`
2. `Skills`

推荐样式：

1. 以 chip/tag 形式展示
2. 运行中高亮当前活跃对象
3. 已完成对象降低强调

### 6.3 当前步骤区

用于高亮当前活跃阶段，只允许有一个主步骤。

示例：

1. `当前阶段：重写解析脚本`
2. `当前动作：正在运行 Bash`

### 6.4 滚动日志区

用于展示执行历史事件流：

1. 限高
2. 内部滚动
3. 默认跟随到底部
4. 用户手动上滑后暂停自动跟随

## 7. 面板状态机

### 7.1 状态定义

执行状态面板存在以下状态：

1. `hidden`
   - 当前没有执行中的 AI 任务
2. `running`
   - AI 正在执行，尚未开始正式输出
3. `failed`
   - 执行中断，且没有正式输出
4. `closing`
   - 已收到正式输出首个文本 token，面板开始关闭动画
5. `closed`
   - 面板已退出当前回复区域

### 7.2 状态切换

1. `hidden -> running`
   - 首个执行事件到达
2. `running -> failed`
   - 本轮执行终止，且最终没有正式文本输出
3. `running -> closing`
   - assistant 正文首个文本 token 到达
4. `closing -> closed`
   - 关闭动画结束
5. `failed -> hidden`
   - 用户发起下一轮消息，或会话重置

### 7.3 关闭触发规则

正式关闭的唯一推荐触发点：

`assistant 正文首个 text token 到达`

不建议使用以下触发点：

1. 最后一个工具调用完成
2. 最后一个 `skill` 完成
3. 最后一个智能体完成

原因是这些事件后通常仍可能有整理答案、阶段切换或最终组织语言的过程。

## 8. 事件模型

执行状态面板不直接暴露原始思考文本，而是把运行时事件映射为面向用户的执行事件。

### 8.1 需要展示的事件类型

1. `agent_spawned`
   - 已调度智能体
2. `agent_finished`
   - 智能体完成
3. `skill_loaded`
   - 已加载 skill
4. `phase_started`
   - 开始阶段
5. `phase_switched`
   - 切换阶段
6. `plan_step_started`
   - 开始执行方案步骤
7. `plan_step_finished`
   - 方案步骤完成
8. `tool_call_started`
   - 工具调用开始
9. `tool_call_finished`
   - 工具调用完成
10. `tool_call_failed`
   - 工具调用失败
11. `result_summary`
   - 阶段性结果摘要

### 8.2 日志示例

1. `已调度智能体：PDF 分析助手`
2. `已调度智能体：脚本修复助手`
3. `已加载 skill：pdf`
4. `已加载 skill：plan-review`
5. `开始阶段：检查 PDF 内容完整性`
6. `执行步骤：验证文件页数`
7. `工具调用：Bash`
8. `工具结果：Pages: 1`
9. `切换阶段：重写解析脚本`

### 8.3 不建议直接展示的内容

1. 冗长原始思维文本
2. 未经清洗的协议层 JSON
3. 对普通用户无意义的底层 SDK 标记
4. 过长工具参数全文

## 9. 交互规则

### 9.1 默认可见规则

1. 进入执行态后立即出现
2. 默认展开
3. 用户不需要点击即可看到内容

### 9.2 高度与滚动规则

推荐桌面端高度：

1. 最大高度：`240px`
2. 极端长日志时内部滚动

自动滚动规则：

1. 默认跟随到底部
2. 用户主动上滑后停止自动跟随
3. 停止自动跟随后显示 `回到最新` 按钮
4. 用户点击 `回到最新` 后恢复自动跟随

### 9.3 多阶段规则

1. 同一时刻只允许存在一个主面板
2. 新阶段开始时，旧阶段不再占据独立大块
3. 当前活跃阶段显示在“当前步骤区”
4. 已结束阶段沉入日志区，保留历史记录

### 9.4 多智能体规则

1. 顶部摘要显示总数
2. 固定参与者区展示所有已调度智能体
3. 日志项必须标记来源智能体
4. 如果并行执行，当前步骤区优先显示用户最关心的主动作

### 9.5 正式输出交接规则

1. 一旦正文开始输出，面板进入 `closing`
2. 执行动画建议使用：
   - `opacity` 淡出
   - `max-height` 收起
3. 关闭时长建议：
   - `120ms - 180ms`
4. 关闭后正文正常接管视觉焦点

### 9.6 失败规则

1. 如果执行失败且没有正文输出，面板不关闭
2. 状态变为 `执行失败`
3. 当前步骤区变为失败摘要
4. 日志区保留最后失败事件

## 10. 文案规范

### 10.1 主文案

建议替换现有模糊文案：

1. `思考中...` -> `AI 正在执行任务`
2. `思考过程` -> `执行记录`

### 10.2 状态文案

1. `执行中`
2. `执行已完成`
3. `执行失败`

### 10.3 摘要文案

1. `已调度 2 个智能体 · 已加载 3 个 skills`
2. `当前步骤：检查 PDF 内容结构`
3. `当前动作：正在运行 Bash`

### 10.4 失败文案

1. `执行中断，请查看日志`
2. `工具调用失败`
3. `未获得有效结果`

## 11. 视觉规范

### 11.1 总体方向

面板视觉延续现有项目的克制、轻玻璃感和低噪声风格，不做终端式重视觉，不做高饱和 AI 面板风。

### 11.2 布局规则

1. 宽度与消息容器一致
2. 位置固定在当前 assistant 回复区域上方
3. 不做悬浮侧栏
4. 不做独立弹窗

### 11.3 层级规则

1. 标题区最强
2. 当前步骤区次强
3. 参与者区固定但弱于当前步骤
4. 历史日志弱于当前步骤但强于背景

### 12.2 新的视图模型

建议在现有 `TimelineSegment` 之上新增统一视图模型：

```ts
interface ExecutionPanelState {
  visible: boolean
  status: 'running' | 'failed' | 'closing' | 'closed'
  title: string
  summary: string
  currentStep?: string
  currentAction?: string
  agents: ExecutionActor[]
  skills: ExecutionActor[]
  logs: ExecutionLogItem[]
  autoFollow: boolean
}

interface ExecutionActor {
  id: string
  name: string
  status: 'running' | 'completed' | 'failed'
}

interface ExecutionLogItem {
  id: string
  type: 'agent' | 'skill' | 'phase' | 'plan_step' | 'tool' | 'result'
  title: string
  detail?: string
  status: 'running' | 'completed' | 'failed'
  timestamp: string
  sourceLabel?: string
}
```

### 12.3 组件边界建议

建议新增：

1. `ExecutionPanel`
2. `ExecutionPanelHeader`
3. `ExecutionParticipants`
4. `ExecutionCurrentStep`
5. `ExecutionLogList`

建议降级或收口：

1. `ThoughtProcess`
   - 不再作为运行中主卡片，转为内部日志渲染能力或被部分替代
2. `SkillCard`
   - 不再以独立卡片形式出现在运行态主流程
3. `SubAgentCard`
   - 不再以独立卡片形式出现在运行态主流程

## 13. 实施阶段

### 阶段 1：统一运行时视图模型

目标：

1. 基于现有 `thoughts + timelineSegments + toolStatus` 生成 `ExecutionPanelState`
2. 不改现有最终正文逻辑

产出：

1. 新的事件映射函数
2. 新的面板状态聚合函数

### 阶段 2：替换运行态 UI

目标：

1. 用 `ExecutionPanel` 替换当前运行中的散卡片组合
2. 接入固定摘要、参与者区、当前步骤区和滚动日志区

产出：

1. 运行态统一执行面板
2. 自动跟随滚动
3. 手动上滑暂停自动跟随

### 阶段 3：输出交接与失败态

目标：

1. 接入“正文首 token 到达后关闭”
2. 接入失败态保留逻辑
3. 调整完成态与错误态的视觉反馈

产出：

1. 完整的执行态 -> 输出态切换
2. 稳定的失败态收口

## 14. 验收标准

以下条件全部满足才算完成：

1. 执行期间，面板默认展开且固定显示。
2. 顶部固定区持续显示：
   - 已调度智能体数量
   - 已加载 skills 数量
   - 当前步骤
3. 长日志不会把消息流无限撑高。
4. 新日志到来时，默认滚动到最新。
5. 用户手动查看旧日志时，不会被强制拖回底部。
6. 多阶段执行时，界面仍只有一个主面板。
7. 多智能体执行时，用户能明确看到参与者和来源。
8. 正式输出开始时，执行面板自动关闭。
9. 如果本轮失败且没有正文，面板保留并显示失败态。

## 15. 风险与注意事项

1. **不要把完整原始推理直接暴露给普通用户**
   - 会造成噪声过高、长度失控和认知负担增加。
2. **不要保留运行态多卡片并列方案**
   - 这会继续破坏统一心智模型。
3. **不要把关闭时机绑定到工具完成**
   - 会导致面板关闭过早。
4. **不要让参与者信息随滚动区一起消失**
   - 用户最关心的“谁在做事”必须固定可见。

## 16. 评审结论请求

本设计确认以下核心决策：

1. 执行期间固定显示统一执行状态面板
2. 面板默认展开
3. 固定展示 `Agents` 与 `Skills`
4. 日志区内部滚动并自动跟随最新
5. 正式输出开始时自动关闭

如以上 5 点无异议，可进入实施计划拆解阶段。
