# Kite 统一工作台首页改造设计

- 日期：2026-03-23
- 状态：待评审
- 适用范围：`HomePage`、`SpacePage`、左侧导航、聊天空状态、tabs、右侧文件/产物面板

## 1. 背景

当前产品把“首页”“空间页”“聊天空状态”拆成了三套叙事：

1. 首页在解释产品、解释空间、引导开始。
2. 进入空间后又切到另一套真正的工作界面。
3. 聊天区内部还有第三套 `Ready to start` 空状态。

结果是首装用户第一眼不知道产品能立刻干什么，也不清楚“空间”与“对话”的关系。

本次设计目标不是继续优化首页文案，而是取消“介绍型首页”，把默认入口收敛成统一工作台。

## 2. 设计目标

### 主目标

`默认页 = 工作台`

用户打开应用后，不应该先进入“空间大厅”或“引导页”，而应该直接进入一个可立即开工的工作台。

### 成功标准

1. 首装且已配置模型的用户，在 30 秒内可以发出第一条消息。
2. 用户第一眼能理解四个区域的职责：
   - 左栏：我在哪
   - 中区：我现在做什么
   - 顶部 tabs：我打开了什么
   - 右栏：我产出了什么
3. 用户不需要先理解“空间”的概念，也能正常开始使用。
4. `skills / agents / commands / workflows` 不再干扰首要任务。

## 3. 外部参考与结论

### 参考来源

1. OpenAI 官方文章：[Designing delightful frontends with GPT-5.4](https://developers.openai.com/blog/designing-delightful-frontends-with-gpt-5-4)
2. OpenAI 帮助中心：[Using projects in ChatGPT](https://help.openai.com/en/articles/10169521-using-projects-in-chatgpt)
3. Anthropic 帮助中心：[How can I create and manage projects](https://support.claude.com/en/articles/9519177-how-can-i-create-and-manage-projects)

### 归纳出的设计结论

1. 首屏主任务应该是“马上开始一次工作”，而不是先学习项目/空间概念。
2. `Project / Space` 更适合作为组织层和上下文层，放在侧边栏与创建流中表达。
3. 引导应该尽量嵌入用户真实要操作的位置，而不是额外做大块解释区。

## 4. 最终产品原则

1. 聊天是入口。
2. 空间是作用域。
3. tabs 是打开内容，不是聊天历史。
4. 右侧文件区是产出面板，不是首页入口。
5. 设置页承接所有高级配置。

一句话定义：

`聊天列表在左，打开内容在上，文件结果在右。`

## 5. 信息架构

### 5.1 系统层

放入设置或独立配置页：

1. 模型配置
2. `skills`
3. `agents`
4. `commands`
5. `workflows`

这些能力保留，但从默认工作台降级，不再占据一级导航和首屏注意力。

### 5.2 工作区层

左侧统一导航负责空间切换和当前空间最近聊天。

### 5.3 工作流层

中区承载聊天、消息流、快捷任务、输入区；tabs 承载当前打开内容；右栏承载文件与产物。

## 6. 最终界面骨架

### 6.1 左栏

左栏只回答两个问题：

1. 我现在在哪个空间
2. 我最近在这个空间做了什么

#### 左栏结构

1. 顶部
   - 当前空间名称
   - 空间切换器
   - `+ 新建空间`
2. 中部
   - `主页`
   - 当前空间卡
   - 当前空间最近聊天列表
   - 其他空间列表
3. 底部
   - `设置`

#### 左栏规则

1. 只展开当前空间的最近聊天。
2. 其他空间默认折叠，只显示名称和状态。
3. 左栏不显示 `skills / agents / commands / workflows`。
4. 左栏不显示完整文件树。

### 6.2 中区

中区是唯一主入口。

#### 空状态

1. 短标题：`直接开始`
2. 短说明：`描述目标，Kite 会产出文件、草稿、代码和步骤。`
3. 快捷任务 chips
4. 大输入框

#### 工作状态

1. 直接显示消息流
2. 输入框固定在底部
3. 不再重复大块欢迎区

### 6.3 顶部 tabs

tabs 只表示“当前打开的工作内容”。

#### tabs 承载内容

1. 当前聊天
2. 计划
3. 文件/代码/HTML/表格预览
4. 搜索或生成结果

#### tabs 规则

1. `新建对话` 由左栏触发，不由 tabs 触发。
2. tabs 不承载聊天历史，不承载空间切换。

### 6.4 右侧文件/产物面板

右栏用于展示结果和上下文，不作为主入口。

#### 默认状态

1. 默认折叠
2. 显示文件/产物图标
3. 可显示数量徽标

#### 展开后内容

1. 最近产物
2. 最近文件
3. 打开空间目录
4. 可选轻量文件树

#### 触发规则

1. 生成新文件后增强提示
2. 用户手动点击时展开
3. 不默认常驻完整文件树

## 7. 三种首页状态

首页不是固定页面，而是同一个工作台在不同状态下的不同表现。

### 7.1 首装用户，未配置模型

#### 目标

让用户立刻知道为什么不能发消息，以及下一步去哪。

#### 表现

1. 左栏正常显示空间结构。
2. 中区仍显示聊天输入区骨架。
3. 输入区位置替换为阻断提示：
   - 标题：`先完成模型配置`
   - 说明：`完成模型配置后，你就可以直接向 Kite 发送任务。`
   - 主按钮：`去设置模型`
4. 快捷任务可以保留，但点击后仍指向模型配置。

#### 原则

不要额外做一整块 onboarding 卡片。阻断说明必须放在“用户本来要发消息的位置”。

### 7.2 已配置模型，但还没开始聊

#### 目标

让用户发出第一条消息。

#### 表现

1. 左栏正常显示空间与聊天结构。
2. 中区显示短标题、短说明、快捷任务、大输入框。
3. 顶部默认只有一个当前聊天 tab。
4. 右栏默认折叠。

### 7.3 已有聊天、持续工作中

#### 目标

保持连续工作流，不做重复欢迎。

#### 表现

1. 左栏高亮当前聊天。
2. 中区直接显示消息流。
3. tabs 显示当前打开内容。
4. 右栏只在有文件结果时提升存在感。

## 8. 首页需要删除的内容

以下内容不应继续作为默认页主体：

1. 首页顶部三块说明卡：
   - `What can I do`
   - `Where should I start`
   - `What should I click now`
2. `Enter Kite` 大卡
3. `3 steps onboarding`
4. 以空间卡片网格作为首页主内容
5. 左栏常驻 `skills / agents / commands / workflows`
6. 首页与空间页两套独立欢迎空状态

## 9. 首页保留但降级的内容

以下内容保留，但只作为次级能力：

1. 空间切换
2. 新建空间
3. 快捷任务
4. 模型配置引导
5. 文件/产物入口
6. tabs 工作台能力

## 10. 页面职责重组

### 10.1 HomePage

`HomePage` 从“介绍型首页”改成“统一工作台壳子”。

#### 新职责

1. 承载左栏
2. 承载中区主工作流
3. 承载顶部 tabs
4. 承载右侧文件/产物入口

### 10.2 SpacePage

`SpacePage` 不再作为另一套独立主界面存在，而是把其工作区能力并入统一工作台。

#### 并入能力

1. 会话切换
2. 聊天区
3. artifact rail
4. canvas

### 10.3 设置页

设置页承接全部高级配置与系统级能力。

## 12. 非目标

本次设计不处理以下内容：

1. 具体视觉稿和像素级样式
2. 完整动效规范
3. 文件树的深度交互细节
4. `skills / agents / commands / workflows` 的配置页详细信息架构

## 13. 风险与约束

1. 如果首页和空间页继续并存两套主要叙事，改造会失败。
2. 如果把高级配置重新塞回左栏，首屏注意力会再次被打散。
3. 如果 tabs 与聊天历史职责不清，用户会混淆“历史对话”和“当前打开内容”。

## 14. 最终结论

这次改造的本质不是调文案，也不是优化 onboarding，而是更换默认入口的产品定义：

`默认页 = 工作台`

用户打开应用时，看到的应该是一个能立即开工的界面，而不是一个要求他先理解空间概念的大厅。
